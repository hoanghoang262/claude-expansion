# Decision: v0.3 Architecture Redesign

**Date:** 2026-03-27
**Status:** accepted

## Context

v0.2 orchestrator được thiết kế với process-first mindset và implicit web/product development bias. Nhiều vấn đề phát hiện qua analysis:
- Pipeline cứng nhắc (mọi task → full clarify→spec→plan→execute)
- Memory system flat, không có STM/LTM distinction
- PM "understand" = understand feature requirements (quá hẹp)
- Phase names bias: CLARIFY (product dev) và IMPLEMENT (code)
- Constraints bị duplicate giữa SKILL.md và action files

## Decisions Made

### D-1: Understanding-first philosophy
**Decision:** Process emerge from understanding, không phải ngược lại.
**Why:** PM tốt không follow pipeline — PM hiểu user + project rồi chọn process phù hợp.
**Consequence:** SKILL.md thinking model phải capture understanding loop, không phải pipeline routing.

### D-2: Phase rename UNDERSTAND/ACT
**Decision:** CLARIFY → UNDERSTAND, IMPLEMENT → ACT.
**Why:** Tên cũ bias về product dev. Tên mới đúng với mọi loại task.
**Consequence:** Stage names trong SKILL.md updated. Action file names (clarify.md, etc.) giữ nguyên.

### D-3: Memory model STM/LTM/Consolidation
**Decision:** Phân biệt rõ working memory (STM) và accumulated understanding (LTM). Consolidation sau bất kỳ completed work loop nào.
**Why:** Não người hoạt động theo model này. Memory = externalized UNDERSTANDING, không phải data storage.
**Consequence:** docs/ structure cần update. Consolidation ritual cần được define.

### D-4: Mode patterns (RESPOND/DIRECT/BUILD)
**Decision:** Ba descriptive patterns, không phải prescriptive pipelines. PM recognizes từ understanding, không phải từ request text.
**Why:** Tránh recreate rigidity ở level khác. Modes = rough guides, not scripts.
**Consequence:** Mode Patterns section trong SKILL.md. Không có action files riêng cho modes.

### D-5: Layer rule (global vs local constraints)
**Decision:** SKILL.md chứa global invariants only. Action files chứa local procedures + local constraints. Không duplicate.
**Why:** Single source of truth per rule. Dễ maintain.
**Consequence:** Removed duplicates từ init.md, execute.md, clarify.md.

### D-6: Adaptive memory structure
**Decision:** docs/ structure không cố định — grows organically theo project type.
**Why:** Web app, research project, learning project cần structures khác nhau.
**Consequence:** Init creates minimal baseline, grows on demand. Templates cần được rethink.

### D-7: Code files vs source files
**Decision:** PM có thể edit documentation files (README, configs) trực tiếp nếu clearly scoped. Chỉ code files (.ts, .py, .go...) phải qua implementer agent.
**Why:** "Fix 3 lines in README" không cần full agent pipeline.
**Consequence:** Hard constraint #1 và #6 updated để reflect distinction này.
